# System Requirements Document (SRD)
## OTL Compiled Object Binary Specification & Compatibility Layout

## 1. Purpose & Core Invariants
This specification defines the binary format for Open Trading Compiled (`.otc`) asset files generated by MarketLab. It provides a highly optimized, immutable, and cross-platform bytecode standard that allows custom formula nodes to be saved, cached, and shared across different workstation installations without exposing or re-parsing the original `.otl` source text.

### Hard Invariants
1. **Zero Runtime Latency:** Loading a pre-compiled `.otc` node must entirely skip front-end lexing, tokenization, and normalization. The engine must hydrate the canvas port interface and spawn the background pipeline execution threads directly from the serialized data structure.
2. **Fail-Fast Platform Bounds:** To prevent silent memory faults or execution crashes between drifting application version installs, every binary file must assert strict magic-byte and compiler generation boundaries at the very beginning of its byte vector array before any asset data is loaded.

---

## 2. Binary Layout Specification
An `.otc` file is structured as a contiguous, byte-aligned data stream split into four precise segments:

### Segment 1: Fixed Magic Header (8 Bytes)
* **Bytes 0–3:** `[0x4F, 0x54, 0x43, 0x42]` — Strict ASCII identifier signature for `OTCB` (Open Trading Compiled Bytecode).
* **Bytes 4–7:** Structural platform alignment and endianness verification padding.

### Segment 2: Compatibility & Engine Metadata (24 Bytes)
* **Bytes 8–9 (u16):** Major Application Version (e.g., `1` for v1.x).
* **Bytes 10–11 (u16):** Minor Application Version (e.g., `4` for v1.4.x).
* **Bytes 12–15 (u32):** **Engine Generation Target Number**. This counter increments whenever there is a breaking change to the underlying mathematical execution engine (`VectorTA` math block layout or graph orchestration runtime).
* **Bytes 16–23 (u64):** Feature Flag Bitmask. Tracks required execution extensions (e.g., Bit 0 = Scalar Constants, Bit 1 = Complex Multivector/Geometric Algebra).

### Segment 3: Node Interface Manifest (Variable Length)
* **Structure:** A compact binary JSON (BSON) or serialized dictionary payload containing the structural signatures extracted during the compile phase.
* **Contents:** Explicit lists of input sockets (names, types like `float`/`int`/`string`, fallback defaults) and output sockets (named AOV streams like `upper_band`, `baseline`, `lower_band`). This data block is used by `registry.rs` to reconstruct the visual node interface on the canvas instantly.

### Segment 4: Mathematical Execution Bytecode (Variable Length)
* **Structure:** The serialized Abstract Syntax Tree (AST) or intermediate representation (IR) expression matrices.
* **Processing:** Read directly by `compile_script_multi_with_context` to instantly generate thread-safe, high-performance `MultiSeriesClosure` instances.

## 3. System Task Implementations for Cursor

### Task 1: Binary Serialization Pipeline (`binary.rs`)
Create the core encoding and decoding asset handling layer.
* **Target File:** `crates/pulsar_marketlab_core/src/orchestration/binary.rs`
* **Mechanics:**
  - Implement `OtcBinaryEncoder::serialize_to_bytes(context: &ScriptCompileContext, manifest: &NodeManifest, bytecode: &[u8]) -> Vec<u8>`. This function must explicitly assemble the 8-byte magic tag, stamp the current application version and engine generation, append the serialized manifest block, and append the bytecode payload.
  - Implement `OtcBinaryDecoder::deserialize_from_bytes(bytes: &[u8]) -> Result<OtcCompiledAsset, OtcError>`. This routine reads the byte vector, validates the file structure against the target system, and returns a fully populated asset struct.

### Task 2: Safety Boundary and Version Verification Pass
Prevent breaking asset states across remote installs.
* **Target File:** `crates/pulsar_marketlab_core/src/orchestration/binary/verifier.rs`
* **Mechanics:**
  - When a binary asset stream is read, extract the `Engine Generation Target Number` (Bytes 12–15).
  - Compare the asset's generation tag directly against the local installation's runtime constant (`const CURRENT_ENGINE_GENERATION: u32`).
  - If the asset generation is greater than the system generation, block the load sequence and return a clean, descriptive execution error to the UI logs: `"Asset compatibility mismatch. The selected node requires MarketLab Engine Generation X, but your installation is running Generation Y. Please update your software."`

### Task 3: OpenUSD Stage Asset Integration Hooks
Connect compiled files directly into the active graph persistence layer.
* **Target Files:** `crates/pulsar_marketlab/src/workspace_state.rs` and `crates/pulsar_marketlab_ui/src/workspace/canvas.rs`
* **Mechanics:**
  - Update the `OtlShader` prim schema configuration to accept an optional file path attribute: `inputs:script_compiled_path`.
  - When a `.otc` file reference is bound to the node card, bypass text compilation completely. Read the binary file directly via the new `OtcBinaryDecoder`, dispatch the interface properties to `sync_otl_shader_ports_from_script` to hydrate the visual layout sockets, and pass the bytecode block directly to the thread execution context loop.

## 4. Verification Test Harness
Add a dedicated, comprehensive integration test block to your compiler test files (`crates/pulsar_marketlab_core/src/orchestration/compiler/tests.rs`) to ensure this entire asset lifecycle works perfectly end-to-end:

```rust
#[cfg(test)]
mod binary_pipeline_tests {
    use super::*;
    use pulsar_marketlab_core::orchestration::binary::{OtcBinaryEncoder, OtcBinaryDecoder, OtcBinaryHeader};

    #[test]
    fn test_otc_binary_roundtrip_and_version_safety() {
        // 1. Simulate mock compiled components
        let mock_manifest_json = r#"{"inputs":[{"name":"source_stream","type":"float"}],"outputs":[{"name":"upper_band","type":"float"}]}"#;
        let mock_bytecode = vec![0xDE, 0xAD, 0xBE, 0xEF]; // Mock AST bytecode instruction array

        // 2. Perform serialization pass using current framework generations
        let current_generation = 2;
        let serialized_bytes = OtcBinaryEncoder::new()
            .with_engine_generation(current_generation)
            .with_manifest(mock_manifest_json)
            .with_bytecode(&mock_bytecode)
            .encode()
            .expect("Failed to serialize OTL node into binary bytecode");

        // Assert magic header tag is accurately written in the primary slots
        assert_eq!(&serialized_bytes[0..4], b"OTCB");

        // 3. Roundtrip Test: Deserialize back to ensure total data preservation
        let decoded_asset = OtcBinaryDecoder::decode(&serialized_bytes)
            .expect("Failed to parse and decode valid OTCB binary stream");
        
        assert_eq!(decoded_asset.manifest_json, mock_manifest_json);
        assert_eq!(decoded_asset.bytecode, mock_bytecode);
        assert_eq!(decoded_asset.header.engine_generation, current_generation);

        // 4. Compatibility Validation Test: Force a validation failure with an older system generation
        let outdated_local_system_generation = 1;
        let validation_result = decoded_asset.header.validate_compatibility(outdated_local_system_generation);
        
        assert!(
            validation_result.is_err(), 
            "The asset pipeline should have rejected a newer engine generation block."
        );
        
        if let Err(err) = validation_result {
            assert!(err.to_string().contains("Asset compatibility mismatch"), 
                "Error output must clearly report generation mismatch constraints.");
        }
    }
}
