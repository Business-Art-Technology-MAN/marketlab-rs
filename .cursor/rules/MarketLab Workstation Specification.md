## 1. System Topology Overview

MarketLab operates on a **Strict Split-Plane Architecture**. System state is completely decoupled into structural declaration (OpenUSD) and mathematical, frame-based array execution (`MarketStage`).

```
 ┌────────────────────────────────────────────────────────────────────────┐
 │                      STRUCTURAL PLANE (OpenUSD)                        │
 │  - Session / User Layer (Active Target Overlays)                       │
 │  - Calculation / Signal Layer (Compiled OTL Topologies)                │
 │  - Base Asset Taxonomy Layer (Read-Only GICS / Tickers)                 │
```

## 2. Structural Layer Stack Specs

The workspace shifts from a flat, text-recomposed single file to a layered OpenUSD composition window (`Layer Stack`). The system evaluates the final stage topology via **LIVRPS** resolution rules.

### 2.1 Layer Hierarchy Boundaries

The stage must load and stack three explicit layer files inside the `WorkspaceContext`:

1. **`session.usda` (Active Edit Target)**: Captures user node canvas transformations, visual parameters (`ui:canvas:pos`), custom node names (`info:user_label`), and manual scalar attribute overrides.
2. **`signals.usda` (Generated/Cached Linkage)**: Stores structural relationship nodes (`OtlTaUberSignal`, `PortfolioIntegrator`) compiled by the graph engine.
3. **`sp500_universe.usda` (Immutable System Asset Base)**: Read-only ledger containing absolute asset identities, exchange MIC codes, and asset category data.

## 3. Node Identity & Stable Path Specification

To protect relationship maps (`rel inputs:sources`) from snapping when a user renames a node inside the GUI, **OpenUSD primitive names must remain fully stable and immutable**.

### 3.1 Naming Protocol Rules

* **Primitive Path**: When a node is dropped onto the canvas canvas, it is assigned an autogenerated, persistent unique string identity (e.g., `/MarketLab/Signals/node_7f89bc`). This path handles all internal routing and compilation passes.
* **User Display Label**: Custom names typed by users (e.g., `EMA_20_Cross`) are written directly onto the primitive as an explicit metadata token attribute:
```usd
custom string info:user_label = "EMA_20_Cross"

```


* **UI Tree Resolution**: The Stage Hierarchy Panel, Param Inspector, and Dope Sheet labels look up `info:user_label`. If empty, they fallback gracefully to displaying the core primitive name string.

## 4. Dual-Representation Timeline & Dope Sheet Engine

Time metrics map bi-directionally between user-facing temporal absolute dates and computer-friendly integer sequencies native to OpenUSD's structure.

### 4.1 Chronological-to-Frame Conversion Map

The engine maintains a static vector index map matching calendar boundaries:

$$\text{Frame } (t) \iff \text{Absolute Date String (YYYY-MM-DD)}$$

* **Timeline Playhead**: Moving the playhead slider updates a universal atomic integer cursor state: `current_frame: usize`.
* **The USD Evaluation Loop**: Scrubbing the playhead executes a synchronous evaluation call:
```rust
let current_time_code = openusd::TimeCode::from_f64(current_frame as f64);
let resolved_attr = usd_attribute.get_at_time(current_time_code);

```


* **In / Out Clipping Vectors**: Visual slider brackets set range bounds `start_frame` and `end_frame`. The core execution engine uses these markers to slide and slice data vectors before invoking stateless processing operations.

## 5. Uniform Visualization Patterns

To eliminate one-off frontend component implementations, the UI view framework assigns visual layouts automatically by parsing the primitive `outputs:*` token signatures defined in your `schema.usda`.

| USD Output Signature | Visual Type Class | Canvas UI Render Mode |
| --- | --- | --- |
| `token outputs:signal` | **Scalar / Condition Flag** | State dot colors (Green = Long, Red = Short, Grey = Neutral) stamped as event markers across the horizontal Dope Sheet track. |
| `float outputs:result` | **Continuous Vector Series** | Real-time continuous line chart layered perfectly on top of the parent asset's main candlestick view. |
| `matrix outputs:weights` | **Multivariate Matrix Layer** | Color-stacked allocation percentage area chart displaying weight rebalancing transitions over time. |

## 7. Operational Implementation Directives for Cursor

When implementing features using this specification, your transformations must rigorously satisfy these compiler constraints:

1. **Enforce Immutability of Paths**: Never write code that transforms a primitive name layout string (`node_XXXX`) when editing node parameter parameters inside the inspector dashboard.
2. **Allocation Isolation**: The `MarketLabGraphEngine::sweep` operation must handle data modifications with **zero heap allocations** inside the framework loop.
3. **Layer Separation**: Ensure any parameters authored during user sessions are explicitly committed to the top `session.usda` edit layer target. Never merge user parameter adjustments down into the base `sp500_universe.usda` resource file.
