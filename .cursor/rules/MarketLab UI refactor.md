---

## 1. Architectural Ground Rules & Performance Contracts

The coding AI **must not** deviate from these rules. Any attempt to introduce real-time mutations or per-frame iterations through an OpenUSD core loop will fail performance criteria.

* **The Split-Plane Constraint:** OpenUSD (`openusd` crate 0.3.0) is strictly for **offline storage, structural definition, and serialization**. It **does not** act as the runtime calculation loop or interactive state engine.
* **Interactivity Target:** The user interface layout, node canvas panning, asset tree expansion, and attribute adjustments must remain decoupled from deep matrix computations to maintain a steady render rate of **$\ge 60\text{ Hz}$**.
* **The Inter-Plane Boundary:** * The UI reads and updates a standard Rust memory model via `Graphy` (`GraphDescription`).
* When the user hits **"Compile"**, `marketlab_blueprint_adapter` maps the Graphy description directly into an in-memory execution plan: `StageGraphSnapshot`.
* **No `openusd::Stage::open` or root layer text-sweeps are permitted during the hot compilation or execution path**.

## 3. Data Flow & Cross-Panel Synchronization Contract

To guarantee full layout coordination without interface lockups, the coding AI must implement a explicit decoupled notification pattern.

```text
[ Visual Node Graph / Slider Interaction ]
                   │
                   ▼
     (Executes state mutations)
                   │
                   ▼
  [ BlueprintEditorPanel Root Entity ] ───(Calls cx.notify())───> [ Listener Repaint Updates ]
                   │                                                        │
                   ├─> Refreshes FinanceOhlcChartPanel                      ├─> Stage Tree Sync
                   └─> Refreshes FinanceWealthChartPanel                     └─> Market Canvas Grid

```

1. **State Modification:** Any manual property slider track adjustment or node line connection change calls `flush_finance_property_inputs()`, writing changes directly out to the central `Graphy` data model.
2. **The Update Broadcast:** Upon compilation wrap, the main `BlueprintEditorPanel` coordinates an active context notification update loop (`cx.notify()`).
3. **Downstream Reception:** The individual sub-panels hook directly into the master editor instance as listeners. When the processing run emits a notification signal, the views pull latest arrays directly from the engine cache (`last_finance_sweep`, `last_finance_asset_previews`) and trigger a structural repaint.

The very bottom tray layout must track engine metrics using specific visualization structures.

* **System Resource Array Allocation:** Format layout text explicitly to show active heap footprints: `System: 22 GB / 17 MB`.
* **Nodal Cache Progress Gauge:** Render a percentage loading bar widget linked directly to the graph evaluation engine's memory cache state: `Nodal Cache Health: [██░░░░░░░░] 10%`.
* **Global Compilation Diagnostic LED:** Render an active graphical indicator block at the right margin:
* **Normal Status:** Solid dark green marker.
* **Amber Evaluation Status:** If a configuration error occurs (e.g., an asset layer path resolution error or an invalid reference loop), flip the marker to flashing bright amber, flag the offending graph nodes visually, and safely fall back calculations to the nearest uncorrupted ancestral layer state to prevent application failure.
